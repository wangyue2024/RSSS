# Rust 高频微观市场仿真系统 (RSSS) - 架构设计总纲 v3.0

## 1. 项目愿景与定位

### 1.1 项目定义
本项目是一个基于 **Rust** 构建的高性能、事件驱动型 **计算金融仿真系统 (Computational Finance Simulation)**。
核心目标是在单机环境下，通过 **混合运行时 (Hybrid Runtime)** 架构，模拟大批量（如 1000+ 个）由 LLM 生成的异构 AI Agent 在微观市场结构下的连续高频博弈，以观察和研究流动性、滑点、价格发现等涌现现象。

### 1.2 核心技术特征

* **混合运行时 (Hybrid Runtime)**: 
  - **Rust (Host)** 负责繁重且需要高性能的模块：撮合引擎、指标重计算、并发调度与状态管理。
  - **Rhai (Guest)** 负责轻量级的代理决策逻辑，允许在不重编译 Rust 代码的情况下热加载和执行成百上千个策略脚本。
* **零拷贝状态分发 (Zero-Copy State Distribution)**: 
  - 利用 `Arc<MarketState>` 与 Rust 的所有权机制，在每个 Tick 将统一的全景市场快照（Market Snapshot）以**零拷贝**的方式分发给 1000 个并发 Agent。极大降低了由于深拷贝带来的内存带宽瓶颈和 Cache Miss。
* **确定性并发 (Deterministic Concurrency)**: 
  - 通过 `rayon` 进行多线程决策计算，同时采用 **Seedable RNG** (基于种子的随机数生成器)。主线程调度时通过固定的全局种子派生出各 Agent RNG 以及全局 Shuffle RNG。保证在相同种子下，多线程并行回测结果 **比特级可复现 (Bit-wise Reproducible)**。
* **全链路定点数引擎 (Fixed-Point Math E2E)**: 
  - 核心计算与存储彻底摒弃 `f64`，全部采用 `i64` 微元 (Micros) 表示。缩放因子统一为 $10^6$。
  - 无论是引擎内部的价格比较，还是 Rhai 脚本层面的决策逻辑，均基于整数，彻底消除 IEEE 754 浮点误差引发的对等一致性故障。
* **影子撤单与幽灵订单 GC (Shadow Cancellation)**: 
  - 撮合引擎的 OrderBook 采用 `BTreeMap<Price, LevelQueue>` 提供 O(K) 的深度快照能力，加上 `HashMap<u64, OrderMeta>` 倒排索引支持 **O(1)** 的撤单。
  - 撤单时仅从 Hash 索引移除并扣除对应档位的总挂单量，不遍历 `VecDeque` 队列。真实撮合时若遇到“幽灵订单”（不在索引内），则直接丢弃。系统自带定期 GC 清理驻留内存。

## 2. 仿真核心生命周期 (The Simulation Loop)

系统在一个长回合 (Session) 中连续运行。主循环定义在 `src/simulation/world.rs` 内。每一帧 (Tick) 严格遵循以下 6 个原子操作阶段：

1. **Pre-Calculation (重计算预处理阶段)**
   * Rust 主线程更新当前 Tick 的历史窗口（如 MA, VWAP, RSI 等技术指标）。
   * 将所有的公开市场数据（包括 L2 盘口，五档深度）封装为只读的 `MarketState` 实例，并包裹进 `Arc` 中准备分发。
2. **Agent Decision (并行决策阶段)**
   * 利用 `rayon::par_iter_mut()` 挂载所有 Agent。
   * **Context**: 注入 `Arc<MarketState>` (市场快照)、`AccountView` (个人账户快照) 和 `Arc<AgentOrderBook>` (个人订单簿状态)。并提供一个空的 `ActionMailbox` 用于收集本次决策。
   * **Execution**: Agent 在隔离作用域 (`Scope`) 中执行 Rhai AST。
3. **Collect Actions (决策收集阶段)**
   * 线程池同步后，主线程统一从各 Agent 的 `ActionMailbox` 回收订单提交动作 (`AgentAction` 列表)，确保订单 ID 分配连续且不冲突。
4. **Deterministic Shuffle (洗牌模拟延迟阶段)**
   * **网路延迟模拟**: 使用 `rand_xoshiro` 全局生成器，通过 Fisher-Yates 洗牌算法，对本 Tick 收集到的所有请求进行完全随机打乱。
   * 这一机制制造了同一 Tick 内的排队先后差异，逼真地还原了高频交易中的**抢单滑点**与**流动性枯竭**风险。并清理上一 Tick 的 fill 记录。
5. **Execution & Settlement (执行与结算阶段)**
   * **串行执行**: 所有被打乱后的 Action 依次进入 `OrderBook`。
   * 前置校验：资金冻结逻辑、可用余额和持仓的校验。同时系统具备防自成交对敲功能 (Self-Trade Prevention)。
   * **引擎触发**: 限价/市价的 Maker/Taker 匹配，生成一系列 `MatchEvent` (Trade, Placed, Cancelled, Rejected 等)。
   * **即时结算**: 消费 `MatchEvent`，实时原子更新买卖双方（Maker & Taker）的 `cash`、`stock`、以及锁定资产量。实现双边手续费扣除及 PnL 重算。
6. **Maintenance & GC (引擎维护期)**
   * 定期（通过配置 `gc_interval`）检查当前由于“影子撤单”导致的幽灵订单数量。
   * 一旦达到阈值，触发 `gc_phantom_orders()` 对 BTreeMap 进行清理，释放冗余内存。

## 3. Rhai 脚本接口规范 (API Reference)

脚本开发要求遵循 **零浮点** 和 **基于 Scope 隐式持久化** 规范。系统不再使用 `custom_memory` 这类繁琐的 Map，而是直接利用 Rhai 的顶层变量生命周期。

### 3.1 状态持久化 (Scope Persistence)
Agent 实例内部持有一个持久的 `rhai::Scope`。
所有在 `fn on_tick()` **外部** 声明的变量都会被自然保留跨 Tick 访问：
```rhai
// 初始化全局变量，整个仿真周期内保持状态
let initialized = false;
let my_stop_loss = 0; // 整数表示

fn on_tick() {
    if !initialized {
        initialized = true;
        // rng 是 Rust 注入的随机数生成器
        my_stop_loss = rng.rand_int(20000, 50000); 
    }
}
```

### 3.2 注入模块与访问

- `market`: `Arc<MarketState>`，获取市场快照。
  - 例如 `market.price`, `market.vwap`, `market.volume` 等。
  - 提供 `market.history_price(idx)` 等函数访问原始时序流。
  - 五档盘口通过扁平化变量 `market.bid_price_0`, `market.bid_vol_0` 访问。
- `account`: `AccountView`，包含 `cash`, `stock`, `avg_cost`, `total_equity`, `realized_pnl`。
- `my_orders`: `Arc<AgentOrderBook>`，跟踪 Agent 个人挂单及状态。
  - 挂单数: `my_orders.pending_count()`
  - 查明细: `my_orders.pending_price(i)`, `my_orders.pending_amount(i)`, `my_orders.pending_remaining(i)`。
  - 成交反馈: `my_orders.fill_count()`, `my_orders.fill_price(i)`, `my_orders.fill_amount(i)`。
- `orders`: `ActionMailbox`，操作句柄。
  - `orders.submit_limit_buy(price, amount)` (注意 price 必须是 micros 整数)。
  - `orders.submit_market_sell(amount)`。
  - `orders.submit_cancel(order_id)`。

### 3.3 数学函数库 (`math`)
原生底层加速：
- `arr_sum(arr)`, `arr_mean(arr)`, `arr_std_dev(arr)`, `arr_slope(arr)` (最小二乘法斜率)。
- 所有入参接收数组，内部截断后计算，输出 `i64` 微元。

## 4. 经济模型

### 4.1 手续费 (Friction)
- 通过 `SimConfig.fee_rate_bps` 设定全局万分之一基点手续费。
- 结算时：`fee = (price * volume * fee_rate_bps) / 10000`，由买卖双边 (Maker, Taker) 同时承担。

### 4.2 资金与持仓监控
系统通过锁定期机制保障不发生超卖超买：
- **买单委托 (Bid Placed)**: 实时锁定对应 `cash` = `price * volume`。
- **卖单委托 (Ask Placed)**: 实时锁定挂出的 `stock`。
一旦被拒 (Rejected) 或 撤单 (Cancelled)，锁定金额立即退回。
